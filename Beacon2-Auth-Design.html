1. Purpose & Scope

This document defines the design for the Users, Roles, Privileges and Authentication module of Beacon2 — the modernised replacement for the Beacon u3a management system.

Beacon2 serves many independent u3a organisations. Each u3a must feel as though they are the only user of the system — their data and configuration are completely isolated from all others. This principle is called multi-tenancy and it underpins every decision in this module.

Scope of this document: Login and session management; user accounts; roles; privileges; role assignment to users; and the system-level tier for creating and managing u3a tenants. Data migration from Beacon is noted but detailed separately.

2. Key Design Principles

  • Tenant isolation first. Every database query that touches u3a data must be scoped to a tenant ID. There must be no way for one u3a's data to leak to another.
  • Roles are per-tenant. Role names and their privilege sets are fully configurable by each u3a administrator. Default roles are seeded at tenant creation but are not fixed.
  • Roles are additive. A user may hold multiple roles. Their effective privileges are the union of all roles assigned to them.
  • Privilege to view is prerequisite. A user cannot create, change, or delete a resource unless they also have the View privilege for that resource.
  • Minimum privilege principle. The system should guide administrators to assign only the privileges necessary for a role, not the maximum available.
  • Committee roles are flagged. Roles can be marked as "committee roles" — this flag has reporting and display uses but does not change how privileges work.
  • System-level functions are separate. The ability to create a new u3a tenant, suspend a tenant, or access cross-tenant data belongs to a distinct system-level tier that is never exposed within a u3a's own interface.

4. Multi-tenancy Architecture

Beacon2 uses a shared database, separate schemas approach, sometimes called the "schema-per-tenant" pattern in PostgreSQL.

  • Each u3a gets its own PostgreSQL schema (e.g. u3a_oxfordshire). All of that u3a's tables live in their schema.
  • A shared public schema holds system-level tables: the list of tenants, system administrator accounts, and global configuration.
  • The application sets search_path to the correct tenant schema at the start of each request, based on the authenticated user's tenant ID embedded in their JWT.
  • This means there is no need for a tenant_id column on every table — the schema boundary enforces isolation at the database level.
Important: Every API endpoint must validate that the tenant ID in the JWT matches the resource being requested. This must be enforced at the middleware layer, not left to individual route handlers.

Tenant lifecycle (system-level)

When a new u3a joins Beacon2, a system administrator performs these steps:

  1. Creates a new tenant record in the public.tenants table.
  2. Runs a migration to create the u3a's schema and seed its tables.
  3. Seeds the five default roles (see Section 6) and their default privileges.
  4. Creates the first administrator user account for that u3a.

5. Access Tiers

Beacon2 has three distinct access tiers. Users at a higher tier cannot be created or managed from within a lower tier's interface.

TierWhoWhat they can do
System Beacon2 platform operators Create / suspend / delete u3a tenants; view cross-tenant diagnostics; manage system-level configuration; create other system admins
u3a Admin Site Administrator within a u3a Manage all users, roles and privileges for their u3a; access all modules they hold privileges for; configure u3a-level settings
Member / Officer Any user within a u3a Access only the parts of the system their assigned roles permit; may hold one or more roles
Note: The System tier interface is a separate, internal-only application (or at minimum a separate route prefix, /system, protected by IP allowlist in addition to auth). u3a users never see system-tier screens.

6. Roles & Privileges Model

Default Roles (seeded at tenant creation)

When a new u3a is set up, the following five roles are created automatically. All are fully editable — name, committee flag, and privilege set.

Default Role NameCommittee RoleNotes
AdministrationNoBroad administrative access; typically the Site Administrator
Group LeadersNoAccess scoped to groups the user leads
Groups CoordinatorNoCoordinates across all groups
Membership SecretaryYesManages member records and renewals
TreasurerYesManages financial records

How roles work

  • A role has a name, a committee flag, optional notes, and a set of privilege assignments.
  • A user is assigned one or more roles. Their effective access is the union of all privilege assignments across all their roles.
  • Roles are scoped to a tenant — a role in one u3a has no meaning in another.
  • The "Administration" role (or equivalent) should have access to the Role Record and Users List privileges so it can manage roles and users.

Group-scoped privileges

Several privileges apply differently depending on the user's relationship to a group:

  • all — can access records for any group
  • as leader — can only access groups where they are the leader
  • as member — can only access groups where they are a participant

These are stored as separate privilege entries, not as a modifier on a single entry.

8. Database Schema

The following Prisma-style schema covers the auth module. Tables in the public schema are prefixed with sys_; tables in the per-tenant schema have no prefix.

System-level tables (public schema)

// public schema

model sys_tenant {
  id          String   @id @default(cuid())
  name        String                         // e.g. "Oxfordshire u3a"
  slug        String   @unique               // e.g. "oxfordshire" — used as schema name
  active      Boolean  @default(true)
  created_at  DateTime @default(now())
  updated_at  DateTime @updatedAt
}

model sys_admin {
  id            String   @id @default(cuid())
  email         String   @unique
  password_hash String
  name          String
  active        Boolean  @default(true)
  created_at    DateTime @default(now())
  last_login    DateTime?
}

Per-tenant tables (u3a schema)

// per-tenant schema (e.g. u3a_oxfordshire)

model user {
  id            String   @id @default(cuid())
  email         String   @unique
  password_hash String?                      // null = no login access
  name          String
  active        Boolean  @default(true)
  created_at    DateTime @default(now())
  updated_at    DateTime @updatedAt
  last_login    DateTime?
  member_id     String?                      // link to member record (future module)

  user_roles    user_role[]
  refresh_tokens refresh_token[]
}

model role {
  id             String   @id @default(cuid())
  name           String
  is_committee   Boolean  @default(false)
  notes          String?
  created_at     DateTime @default(now())
  updated_at     DateTime @updatedAt

  user_roles     user_role[]
  privileges     role_privilege[]
}

model user_role {
  id         String   @id @default(cuid())
  user       user     @relation(fields: [user_id], references: [id])
  user_id    String
  role       role     @relation(fields: [role_id], references: [id])
  role_id    String
  assigned_at DateTime @default(now())

  @@unique([user_id, role_id])
}

model privilege_resource {
  id       String   @id @default(cuid())
  code     String   @unique        // e.g. "finance:transactions"
  label    String                  // e.g. "Finance: transactions"
  actions  String[]                // allowed actions e.g. ["view","create","change","delete"]
  // Note: actions lists what is POSSIBLE for this resource, not what is granted
}

model role_privilege {
  id          String   @id @default(cuid())
  role        role     @relation(fields: [role_id], references: [id])
  role_id     String
  resource    privilege_resource @relation(fields: [resource_id], references: [id])
  resource_id String
  action      String              // "view" | "create" | "change" | "delete" | "download"
                                  // | "send" | "reconcile" | "renew" | "lapse" | "all" etc.
  @@unique([role_id, resource_id, action])
}

model refresh_token {
  id          String   @id @default(cuid())
  user        user     @relation(fields: [user_id], references: [id])
  user_id     String
  token_hash  String   @unique
  expires_at  DateTime
  revoked     Boolean  @default(false)
  created_at  DateTime @default(now())
}

9. Authentication Flow

Login

  1. User submits email + password to POST /auth/login.
  2. Server looks up user by email within the correct tenant schema (tenant is identified by subdomain or login-page context).
  3. Server verifies password hash with bcrypt.
  4. On success, server issues:
    • A short-lived access token (JWT, 15 minutes), containing: user ID, tenant slug, and computed effective privileges.
    • A long-lived refresh token (opaque string, 30 days), stored as an httpOnly cookie and its hash saved in the database.
  5. The access token is returned in the response body. The frontend stores it in memory only (not localStorage).

Authorisation on each request

  1. Frontend sends the access token as a Bearer header.
  2. API middleware validates the JWT signature and expiry.
  3. Middleware extracts the tenant slug and sets the PostgreSQL search_path to the correct tenant schema.
  4. Route handler checks the required privilege against the token's privilege set.

Token refresh

  1. When the access token expires, the frontend calls POST /auth/refresh.
  2. The server reads the refresh token from the httpOnly cookie, validates it against the database, and issues a new access token.
  3. Refresh tokens are rotated on each use (old one revoked, new one issued).

Logout

  1. Frontend calls POST /auth/logout.
  2. Server revokes the refresh token in Redis and the database.
  3. Frontend discards the access token from memory.
Privilege embedding in JWT: Computing privileges on every request would require multiple database joins. Instead, the effective privilege set is computed at login and embedded in the JWT. When an administrator changes a role's privileges, affected users' sessions are invalidated via Redis so they must re-login and receive a fresh token.

11. Data Migration Notes

A full migration design is out of scope for this document, but the following points are relevant to the auth module specifically:

  • Beacon user passwords cannot be migrated directly (they are hashed with an unknown algorithm and the plaintext is unavailable). The recommended approach is to force a password reset for all users on first login to Beacon2.
  • Roles and their privilege assignments can be migrated by reading the Beacon configuration and mapping to the Beacon2 privilege resource codes defined in Section 7.
  • User–role assignments can be migrated directly once roles are in place.
  • The migration tool should be a standalone script that connects to the Beacon data export and populates the correct tenant schema in Beacon2.
